Code Formatierung¶

Wir folgen den “Google’s Java Style Guidelines <https://google.github.io/styleguide/javaguide.html>” mit ein paar Änderungen und Modifikationen, die in diesem Artikel beschrieben werden.

Tipp

Du kannst unsere Code Formatierungsvorlagen für Eclipse oder IntelliJ IDEA benutzen, um den Code durch deine Entwicklungsumgebung(IDE) korrekt Formatieren zu lassen. Unter Vorbereitung für die Entwicklung findest du weitere Informationen.

  • Zeilenenden

    • Bei der Benutzung von committing (\n), verwende die UNIX Zeilenende

      • Windows Benutzer von Git können “git config –global core.autocrlf true” ausführen, um diese automatisch von Git konvertieren zu lassen.

  • Spaltenbreite

    • 80 für Javadocs

    • 150 für Code

    • Zögern Sie nicht dies Anzupassen, wenn es die Lesbarkeit verbessert.

  • Einzug

    • Verwende 4 Leerzeichen zum Einrücken, nicht 2

  • Leerzeilen

    • Erstelle eine Leerzeile vor dem ersten Eintrag einer Klasse, Interface, Enum, usw. (in der Regel nach class Example {) genauso nach dem letzten Eintrag

  • Datei Kopf

    • Der Datei Kopf muss den Lizenztext vom Projekt enthalten. Verwende den licenseFormat Gradle Befehl um dies automatisch durchführen zu lassen.

  • Importiert

    • Imports müssen in der folgenden Reihenfolge gruppiert werden, wobei die Gruppen durch eine Leerzeile getrennt werden

      • Statische Imports

      • Alle anderen Imports

      • java Imports

      • javax Imports

    • Dies unterscheidet sich von Googles Stil, bei dem die Imports nicht nach der obersten Gruppe, sondern alle gemeinsam zusammengefasst werden.

  • Ausnahmen

    • Wenn “Exceptions” ignoriert werden soll, nenne die Variable “ignored”

  • Zugriff auf Felder

    • Befehle registrieren

  • Javadocs

    • Verwende kein @author

    • Umklammere zusätzliche Paragraphen mit <p> und </p>

    • Schreibe das erste Zeichen in jeder Beschreibung. In der Regel @param name Player to affect. Verwende keine Punkte

Code Konventionen¶

  • Verwende / Plugin/optional/Index statt in der API Rückgabe null

  • Methoden die null akzeptieren müssen mit @Nullable ( javax.*) kommentiert werden. Alle Methoden und Parameter sind normalerweise standardmäßig @Nunnull.

  • Verwende Google Preconditions um Argumente und “null” zu überprüfen.

package com.example.java;

import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

import java.util.Random;
import java.util.Optional;

public class Example {

    private static final Logger log = LoggerFactory.getLogger(Example.class);
    private static final Random random = new Random();
    private final String id = "test";

    /**
     * Returns an identifier approximately half of the time.
     *
     * <p>A static instance of {@link Random} is used to calculate the
     * outcome with a 50% chance.</p>
     *
     * @return The ID, if available
     */
    public Optional<String> resolveId() {
        log.info("ID requested");

        if (random.nextBoolean()) {
            return Optional.of(this.id);
        } else {
            return Optional.empty();
        }
    }

    /**
     * Returns an identifier approximately half of the time.
     *
     * <p>A static instance of {@link Random} is used to calculate the
     * outcome with a 50% chance. If the outcome is to not return the ID,
     * the given fallback ID is returned.</p>
     *
     * @param fallback A fallback name to return
     * @return The ID half of the time, the given fallback the other half
     */
    public String resolveId(String fallback) {
        return resolveId().orElse(fallback);
    }

}
